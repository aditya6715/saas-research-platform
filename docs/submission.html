Motivation & Key Findings Page 2 of 6

Why This Was Built

Composio turns SaaS apps into tools that AI agents can call. Before a single line of integration code is written, a researcher must answer a fixed set of questions for every target app: how does auth work, is access self-serve or gated, what is the API surface, does an MCP server already exist?

For one app, this takes 20–40 minutes of focused research. For 100 apps, that is a full week of human research time per batch — and the same work repeats every quarter as APIs change, pricing shifts, and the MCP ecosystem grows.

The Core Insight

The research task is structured and repeatable. Every app needs the same fields answered using the same sources. This is exactly what AI agents excel at — decomposable into sub-tasks, each with a verifiable output, producing a structured record. The challenge is not extraction, it is reliable extraction at scale.

Why Accuracy Matters More Than Speed

Every wrong data point has a downstream cost. An incorrect auth method means a broken integration in production. An incorrect "Self-Serve" classification means an engineer spends a week on an app that requires a $50k enterprise contract. An undetected MCP server means Composio duplicates community work. This is a trust-sensitive data product — reliability is the primary engineering requirement.

The Agent Pipeline Page 3 of 6

What Was Built — Multi-Agent LangGraph Pipeline

Ten specialized AI agents orchestrated by LangGraph. Each agent has a single responsibility, a Pydantic output schema (no free-form text), and a retry budget. Not a monolithic "do everything" prompt — a production pipeline.

1
🔍
Documentation Finder
Two Tavily web searches with different query formulations. URL scoring: domain match +0.40, /docs path +0.30, cross-query agreement +0.10. Score <0.50 → flag for human review.
✓ Fully automated
2
📄
Documentation Parser
Firecrawl extracts clean markdown. Falls back to Browser Use + Playwright for JS-rendered/bot-protected pages. Crawls up to 20 subpages. Chunks into 8k-token segments with 200-token overlaps.
⚑ Human: 3 apps had login-walled docs
3
⚡
Parallel Extraction — 4 agents running simultaneously
Auth Extractor (GPT-4o, typed schema, 10 valid methods + OAuth flows) · API Analyzer (REST/GraphQL/gRPC/WebSocket/Webhook, base URL, versioning, rate limits, OpenAPI URL) · Dev Portal Agent (Browser Use navigates signup flow, screenshot as evidence) · MCP Detector (3 searches, Official/Community/In-Progress/None classification)
⚑ Human: Salesforce + WhatsApp access model corrected
4
🔗
Evidence Collector
Creates one sourced Evidence record per non-null field. Validates source URLs (HTTPS, reachable). Fields with no evidence get confidence=0.0 and a warning log. Full audit trail.
✓ Fully automated
5
✅
Verification Agent — Blind Second Pass
Does NOT read extraction pass outputs. Uses different doc chunks + adversarial prompt framing. Compares Pass A vs Pass B field-by-field. Disagreement → Tiebreaker Agent (GPT-4o selects better-supported value with one-sentence reasoning).
⚑ Human: 12 records reviewed, 2 corrected
6
📊
Confidence Scorer + Buildability Calculator
Deterministic formulas — no LLM. Field score = evidence_confidence × agreement_bonus × browser_bonus. App score = weighted average (auth 25%, api 20%, access 20%). Verdicts via priority-ordered rule engine. Apps below 0.85 flagged for human review.
✓ Fully automated
7
💡
Pattern Discovery Engine + Report Generator
SQL aggregates over all 100 verified records. Computes MCP gap, easy wins, category breakdowns. Jinja2 renders self-contained 500KB HTML — Chart.js inlined, zero CDN deps, works offline and in Brave.
✓ Fully automated

⚠ Honest Assessment of Both Errors

Both errors were in the conservative direction (over-blocking rather than over-promising). Salesforce has a free Developer Edition that gives self-serve API access — the agent focused on production orgs which are gated. WhatsApp's Cloud API offers a free test phone number via Meta's developer portal — the agent saw "verification required" and stopped there.

This is correct behaviour for a trust-sensitive data product: when portal content is ambiguous, err toward caution. The human review workflow exists precisely to catch and correct these conservative errors. The system flagged both apps for review — it just needed a human to resolve the ambiguity.

Engineering — Architecture & Tech Stack Page 5 of 6

Engineering Decisions — Every Choice Justified

ComponentTechnologyWhy This, Not Something Else
Agent Orchestration LangGraph Explicit directed graph — every state transition visible and debuggable. Conditional edges (branch on confidence score, skip tiebreaker if agents agree). LangChain AgentExecutor is a black box; LangGraph is not.
Web Crawling Firecrawl Documentation is overwhelmingly JavaScript-rendered SPAs (Next.js, Docusaurus). BeautifulSoup returns empty. Firecrawl handles rendering, strips nav/footer noise, returns clean markdown. LLM gets better input → better extraction.
Browser Automation Browser Use + Playwright Access model classification requires navigating a developer portal dynamically — "does clicking Get API Key lead to an instant form or Contact Sales?". Browser Use provides LLM-guided navigation. Raw Playwright for screenshots.
LLM GPT-4o / GPT-4o-mini GPT-4o for extraction/verification (best structured output accuracy). GPT-4o-mini for classification tasks (8× cheaper, sufficient for simpler decisions). Structured outputs via Pydantic — no free-form text parsing.
Database SQLite WAL Zero ops overhead. Single file (copy, share, inspect with any viewer). WAL mode handles concurrent writes from 5 parallel agents. Migrate to Postgres only when multi-user is needed. Right tool for the scale.
Output Format Self-contained HTML No server, no build step, no npm. Email it, host on GitHub Pages, open offline. All interactivity (charts, search, dark mode) achievable with inlined Chart.js + vanilla JS. A React app would add unnecessary complexity.
Buildability Rules Deterministic Python Verdicts must be reproducible and auditable. An LLM verdict can change between runs (temperature, model updates). A rule engine gives identical output for identical input, forever — essential for a data product.

How Composio Would Use This in Practice

🗺 Integration Roadmap

Filter the report: Verdict = Fully Buildable AND MCP = None. Get a ranked list of highest-confidence, immediately-actionable integrations. This is a Q3 roadmap input generated in 30 minutes, not a week of research.

🔍 Pre-Engineering Due Diligence

Engineer assigned to build Salesforce integration. Before writing any code: run the pipeline, get auth methods (OAuth 2.0, flows: auth_code + client_credentials), API surface (REST + GraphQL + SOAP), access model (Dev Edition = self-serve), rate limits, OpenAPI URL. Two days of research in one pipeline run.

📡 Market Monitoring

Run monthly. The session diffing system detects when a Gated app adds a self-serve tier, or when a new Official MCP server appears. Ecosystem changes surface automatically — no human monitoring 100 documentation sites.

💼 Enterprise BD Pipeline

The 20 Gated apps are not failed research — they are enterprise sales leads. The system identifies exactly which apps require partnership or contract-sales access, complete with evidence URLs.

📈 Scaling to 500+ Apps

The CSV is the only input. Add more apps, re-run. Queue, caching, retry, and verification all scale linearly. Rate limit backoff handles API pressure. The architecture was designed for scale from the start.

🤝 MCP Gap Prioritization

66% of apps with a public API have no MCP support. Cross-reference with traffic, customer requests, and engineering complexity to get a weighted priority list. This is the most strategically valuable output in the report.

What Makes This Production-Grade

  • Idempotent ingestion — safe to restart after any crash
  • Atomic queue transitions — no orphaned jobs
  • Two-pass verification — not just one LLM call
  • Deterministic buildability rules — reproducible verdicts
  • Full evidence trail — every field has a cited source URL
  • Human review workflow — structured correction path
  • Schema-versioned database — safe migrations, no data loss
  • 24-hour HTTP cache — 8-min re-runs after first crawl
  • 62 tests, CI on every push — ruff + black + unit + integration + Docker
  • Structured LLM outputs — Pydantic schemas, no free-form parsing
  • Confidence scoring — reliability is explicit, not assumed
  • Exponential backoff — handles rate limits without crashing
  • Session history — re-runs create new sessions, old data preserved
  • Zero CDN dependencies — report works offline, in Brave, anywhere
  • Self-contained HTML output — deploy by email, GitHub Pages, or drag-drop
  • $15 per 100-app run — economically viable at scale
