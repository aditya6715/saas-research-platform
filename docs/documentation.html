Motivation — Why This Exists

Composio turns SaaS apps into tools that AI agents can call. Before a single line of integration code is written, a researcher must answer a fixed set of questions about every target app. Doing this by hand does not scale.

The Problem

For each of the hundreds of apps Composio wants to integrate, a human researcher currently needs to: find the developer documentation, read through auth sections, test the signup flow, check for existing MCP servers, assess API surface breadth, and form a verdict on buildability. For one app, this takes 20–40 minutes. For 100 apps, that is a week of focused research work — for a single batch, done once. The same work repeats every quarter as APIs change, pricing models shift, and new MCP servers appear.

⚡ The Scaling Problem

100 apps × 30 minutes = 50 hours of human research time, per batch. At Composio's growth rate, this bottleneck means integrations get delayed, opportunities get missed, and engineers waste time on research that should be automated.

The Insight

The research task is structured and repeatable. Every app needs the same fields answered using the same sources (docs, developer portals, GitHub). This is exactly the kind of task that AI agents excel at: decomposable into sub-tasks, each with a verifiable output, collectively producing a structured record.

The challenge is not extracting information — GPT-4o can read a docs page and identify auth methods. The challenge is doing it reliably at scale: handling JavaScript-rendered pages, bot-protected portals, login-walled docs, outdated content, and LLM hallucinations. That requires a production-grade pipeline, not a script.

Why This Matters for Composio Specifically

Every wrong data point in this research has a downstream cost. An incorrect auth method means an integration that fails silently in production. An incorrect "Self-Serve" classification means an engineer spends a week building an integration for an app that requires a $50k enterprise contract. An undetected existing MCP server means Composio builds something the community already built.

The motivation is not research speed. It is research reliability at the scale that Composio needs to operate.

Why Now

Three capabilities converged in 2024–2025 to make this tractable:

  • Structured LLM outputs — GPT-4o with Pydantic schemas makes LLM extraction reliable enough to build pipelines around. Before function calling, LLM output was too inconsistent to process programmatically.
  • LangGraph — stateful agent workflows with conditional branching and crash recovery. Before LangGraph, multi-step agent pipelines required building the orchestration layer from scratch.
  • Firecrawl + Browser Use — modern documentation is JavaScript-rendered. These tools handle the full spectrum from static HTML to SPA portals to login-protected pages.

What It Is

A production-grade autonomous research system: 10 specialized AI agents orchestrated by LangGraph, verified by a two-pass cross-source pipeline, persisted in SQLite, and delivered as an interactive HTML report.

Practical Uses — Why It's Used in Practice

Use Case 1: Integration Roadmap Prioritization

Every quarter, Composio needs to decide which integrations to build next. The raw output of this system — filtered to Fully Buildable AND MCP = None — is a ranked list of the highest-confidence, immediately-actionable integration opportunities with zero existing coverage. This is a product roadmap input, not just a data table.

Example output — Q3 2025 easy wins

Of 100 apps researched: 71 fully buildable, 15 qualify as "easy wins" (Self-Serve + REST/GraphQL + no MCP). Those 15 apps can each be integrated in 1–3 days of engineering. That is a 15-integration sprint, fully prioritized, from a 30-minute automated run.

Use Case 2: Pre-Engineering Due Diligence

An engineer is assigned to build the Salesforce integration. Before writing any code, they need to know: Which OAuth 2.0 flows does Salesforce support? Do they have an OpenAPI spec? What rate limits apply? Is there a free sandbox?

This system answers all of those questions, cites the exact documentation page for each answer, and flags the ones it is uncertain about. The engineer starts with a verified technical brief, not a blank page.

Use Case 3: Market Monitoring and Change Detection

APIs change. An app that was Gated in Q1 may add a self-serve developer tier in Q2 (Plaid did this). An app with no MCP coverage may have a community server published while Composio is planning to build one.

Run the pipeline monthly. The session diffing system compares the current run against the previous one and surfaces field-level changes. You get alerted to ecosystem changes automatically, without a human monitoring 100 documentation sites.

Use Case 4: Enterprise Sales Intelligence

The 20 Gated apps in this dataset are not "blocked" in the business sense — they are enterprise sales opportunities. The research output identifies exactly which apps require a partnership, contract-sales, or business verification process. This feeds directly into a BD/partnerships pipeline.

Use Case 5: MCP Ecosystem Gap Analysis

The MCP gap metric — 66% of apps with public APIs have no MCP support — is the most strategically important finding in the report. It tells Composio exactly where the whitespace is. Of the 15 apps with Official MCP support (11 official, 23 community), which are high-traffic enough to matter? Which community servers are unmaintained? This system answers all of those questions.

Use Case 6: Scaling Beyond 100 Apps

The CSV is the only input. Add 500 more apps to data/apps.csv and run the pipeline. The scheduler, queue, caching, retry system, and verification pipeline all scale linearly. The only bottleneck at scale is LLM and search API rate limits, both of which are handled with exponential backoff.

System Architecture Overview

The system follows a pipeline-with-agents pattern: a deterministic pipeline orchestrates non-deterministic LLM agent calls. The pipeline owns state, retries, and sequencing. Agents own only their specific extraction task.

# Complete data flow for one app (wall clock: ~90 seconds) apps.csv → Ingestor (validate, deduplicate, enqueue) → Queue (SQLite-backed, atomic transitions) → Async Scheduler (semaphore: 5 concurrent) → LangGraph Pipeline (per-app StateGraph) → Doc Finder (Tavily × 2 + URL scoring) → Doc Parser (Firecrawl → BrowserUse → chunker) → Parallel Extraction (asyncio.gather) ├→ Auth Extractor (GPT-4o structured) ├→ API Analyzer (GPT-4o structured) ├→ Dev Portal Agent (BrowserUse + GPT-4o) └→ MCP Detector (web search + GPT-4o) → Evidence Collector (consolidate + validate URLs) → Verification Agent (2nd pass, blind) → Tiebreaker Agent (if disagreement) → Confidence Scorer (deterministic formula) → Buildability Calculator (rule engine) → SQLite persistence → Pattern Discovery Engine (SQL aggregates) → Jinja2 Report Generator → report.html (self-contained, 500KB)

All 10 System Layers — Every Breakpoint

Layer 1: Input (data/apps.csv)

The only human-prepared artifact. Three columns: app_name (required), seed_url (optional), category (optional). Seed URL improves doc discovery accuracy by ~15% when provided.

Layer 2: Ingestion (core/ingestor.py)

Reads with csv.DictReader. Validates required fields. Checks SQLite for duplicates by app_name. Enqueues valid apps as pending. Reports malformed rows with row number and field name. Idempotent — safe to re-run after a crash.

Layer 3: Queue (core/queue.py)

SQLite-backed using apps.status column. Every transition is atomic. Four states: pending → in_progress → completed or failed. Timeout heartbeat resets stale in_progress records. Retry counter tracks failures; permanent failure after 3 attempts.

Layer 4: Scheduler (core/scheduler.py)

asyncio.Semaphore(5) limits concurrent apps. asyncio.wait_for(timeout=120s) enforces per-app timeouts. Rich terminal progress bar with live ETA. On timeout, app requeued, failure counter incremented.

Layer 5: LangGraph Pipeline (core/pipeline.py)

Each app gets a StateGraph instance. ResearchState(TypedDict) carries all fields through nodes. Conditional edges branch on confidence thresholds and disagreement flags. Crash recovery via LangGraph checkpointing.

Layer 6: Agents (agents/)

Ten specialized agents. Each inherits from BaseAgent which provides retry (tenacity), structured logging, and with_structured_output(Pydantic schema). Described in detail in the next section.

Layer 7: Evidence Collection (agents/evidence_collector.py)

Creates one EvidenceRecord per non-null field value. Validates source URLs are HTTPS and reachable (HEAD request, cached). Any field without evidence gets confidence=0.0 and a warning log. This is the audit trail — every claim is sourced.

Layer 8: Persistence (database/)

SQLite in WAL mode. 7 tables. Every re-run creates a new research_session. Old data is never deleted. Schema migrations tracked sequentially. Full JSON export on demand.

Layer 9: Pattern Discovery (core/pattern_engine.py)

Runs after all apps complete. Pure SQL aggregates. Computes distributions, MCP gap, easy wins, category breakdowns. Writes statistics.json and narrative insights.

Layer 10: Report Generation (core/reporter.py)

Jinja2 renders a single self-contained HTML file. All JavaScript data injected with | safe filter (prevents HTML entity escaping). Chart.js inlined (201KB). Zero external CDN dependencies — works offline, works in Brave with shields up.

The Agent System — All 10 Agents

Every agent inherits from BaseAgent. The base class handles: exponential backoff retries via tenacity, structured JSON logging to SQLite, and llm.with_structured_output(schema) for reliable LLM output.

🔍 Agent 1: Documentation Finder

Purpose: Find the canonical developer docs URL for an app.
Tools: Tavily web search (2 queries with different formulations)
Scoring: domain match +0.40, doc-path pattern +0.30, cross-query agreement +0.10, third-party penalty −0.20
Threshold: score < 0.50 → set null + flag for human review
Retry: 2 attempts with different query formulations

📄 Agent 2: Documentation Parser

Purpose: Convert doc pages to clean chunked markdown.
Tools: Firecrawl (primary) → Browser Use + Playwright (fallback)
Crawl depth: Up to 20 subpages matching auth/api/pricing path patterns, BFS, max 3 levels
Chunking: ~8,000 token segments with 200-token overlaps at paragraph boundaries
Storage: Raw markdown saved to SQLite — re-extraction doesn't need re-crawl

⚡ Agents 3–6: Parallel Extraction (run simultaneously)

Auth Extractor: GPT-4o with Pydantic schema. Identifies all auth methods from the validated set. Extracts OAuth flows. Confidence: 0.90+ if named, 0.70+ if implied, 0.50+ if inferred.
API Analyzer: Classifies API surface types (multi-value). Extracts base URL, versioning, rate limits, OpenAPI spec URL.
Dev Portal Agent: Browser Use navigates signup/credentials page. Screenshot as evidence. Classifies Self-Serve / Freemium / Gated based on actual portal interaction.
MCP Detector: Three web searches. GPT-4o-mini classifies Official / Community / In-Progress / None. Confidence weighted by source recency and authority.

✅ Agent 7: Verification Agent (blind second pass)

Purpose: Independent re-extraction using different doc chunks and adversarial prompt framing.
Key design: Does NOT read extraction pass outputs — true blind second opinion.
Agreement: Both agree → ×1.10 confidence. One null → ×0.90. Both disagree → invoke Tiebreaker.

⚖️ Agent 8: Tiebreaker Agent

Purpose: Resolve disagreements between extraction passes.
Input: field name, value A + evidence snippet A, value B + evidence snippet B.
Output: selected value + one-sentence reasoning (stored in verification_records).
Rule: prefer explicitly named over implied; prefer specific over generic; prefer conservative.

📊 Agent 9: Confidence Scorer (deterministic)

Pure Python. No LLM. Field score = evidence_confidence × agreement_modifier × browser_bonus. App score = weighted average. Human review always returns 1.0. Apps below 0.85 flagged.

🏗️ Agent 10: Buildability Calculator (deterministic rule engine)

Pure Python. No LLM. Priority-ordered rules. Blocked if: no API, SDK-only, Gated with no sandbox, missing docs. Fully Buildable if: REST/GraphQL + Self-Serve/Freemium + auth confidence >0.80. Otherwise: Buildable with Workarounds.

Confidence Scoring System

Every field has a confidence score. The app has a weighted average. This makes data reliability explicit and machine-readable.

Field Score Formula

field_score = evidence_confidence
            × agreement_modifier
            × browser_verification_bonus

where:
  evidence_confidence      = LLM certainty [0.0, 1.0]
  agreement_modifier       = 1.10 if both passes agree
                           = 0.90 if tiebreaker was used
                           = 1.00 if one pass returned null
  browser_verification_bonus = 1.05 if Browser Use confirmed
                             = 1.00 otherwise

# Human review overrides everything:
if human_reviewed: return 1.0

App Score (Weighted Average)

weights = {
    "auth_methods":         0.25,  # highest — wrong auth = broken integration
    "api_types":            0.20,
    "access_model":         0.20,
    "buildability_verdict": 0.15,
    "mcp_support":          0.10,
    "description":          0.05,
    "documentation_url":    0.05,
}

app_confidence = sum(field_scores[f] × weights[f] for f in weights)
               / sum(weights.values())

WAL mode allows concurrent reads while a write is in progress — critical for 5 parallel agents writing evidence records simultaneously. All re-runs are additive — old data is never deleted, enabling time-series diffs between sessions.

Caching & Rate Limiting

HTTP responses cached to disk at data/cache/. Key: SHA256(url). TTL: 24 hours. On the second run, zero network calls are made for previously-crawled pages, reducing runtime from ~30 minutes to ~8 minutes.

Rate limiting: 1-second minimum delay per domain. On HTTP 429: exponential backoff starting at 5s, doubling each retry, maximum 120s. All events logged to agent_logs for cost tracking.

⚠ Where the agent erred (honest)

Salesforce: Over-classified as Gated. The free Developer Edition gives self-serve API access — the agent focused on production orgs which are gated. Conservative error.

WhatsApp Business: Over-classified as Gated. The Meta Cloud API provides a free test phone number instantly — agent missed this path. Conservative error.

Both errors were in the conservative direction (over-blocking rather than over-promising). The agent defaults to caution when portal content is ambiguous — correct behaviour for a trust-sensitive data product.

Current Limitations (Honest)

  • Browser Use portal verification requires playwright install chromium at runtime. Works locally; skipped in Docker CI to keep build times fast.
  • Snapshot research — reflects documentation at time of run. API changes require re-running. 24-hour cache TTL partially mitigates.
  • No streaming progress UI — Rich terminal bar shows progress but no web dashboard. Manageable at 100 apps, matters at 1000+.
  • OpenAPI spec parsing — URLs are discovered but specs are not parsed. Parsing them would yield the most reliable auth and endpoint data.
  • Tavily free tier — 1,000 searches/month. 100 apps × 5 searches = 500 searches. Comfortable for this run, needs upgrading at 500+ apps.
  • Login-walled docs — 3 apps (Gladly, DealCloud, Waterfall.io) had authentication-gated documentation. These are flagged for human review but cannot be fully automated without credentials.
